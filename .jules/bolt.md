## 2024-05-19 - Zustand useWorldState Causes N+1 Re-Renders in List Item Components
**Learning:** `useWorldState()` subscribes to the entire global state object by default. When used in a component that is rendered inside a large list (like `FormSparkline` in a roster table), it completely breaks `React.memo()`. Any unrelated global state change (e.g., ticking the week, treasury changing) will trigger a re-render for every single item in the list, causing massive O(N) performance regressions.
**Action:** In small list components, never call `useWorldState()` directly. Instead, extract exactly what you need using a narrow selector like `useGameStore((s) => s.specificSlice)` to maintain `React.memo` isolation. Only use `useShallow` if the selector returns a newly constructed object or array.

## 2024-06-02 - EntityLink re-rendering optimization
**Learning:** `WarriorLink` and `StableLink` components were using `useWorldState()` to fetch global game state just to resolve names to IDs. This caused them to re-render constantly on every game state change, even when their target warrior/stable wasn't affected. Since these components are used everywhere (e.g. `EventLog`, `WarriorAuditCard`), it created a massive performance bottleneck, triggering re-renders across the app.
**Action:** Replace `useWorldState()` with `useGameStore(useShallow(...))` to only subscribe to the strictly necessary state slices (`player`, `rivals`, `roster`, `graveyard`, `retired`) needed for name resolution.
## 2024-06-03 - Replacing useWorldState with useShallow

**Learning:** When replacing full-state subscriptions (`useWorldState()`) with narrowed `useShallow` selections from `useGameStore`, remember that many game state properties (e.g., `roster`, `insightTokens`, `retired`) are optional and may be undefined. Always preserve or re-implement nullish coalescing fallbacks (e.g., `const safeRoster = roster ?? []`) before calling array methods like `.map` or `.find` to prevent runtime crashes. Additionally, when renaming variables for these safe fallbacks (e.g., `safeRoster`), make sure to update *all* references within the component. Finally, ensure any bound functions or actions retrieved from the store (e.g., `consumeInsightToken`) are included in the shallow selector or retrieved separately to avoid inadvertently causing the component to subscribe to the entire state again.

**Action:** Whenever refactoring `useWorldState()` to `useShallow`, I must specifically check for array operations (like `.find()` or `.map()`) and verify that default values (`?? []`) are applied *before* the operations. I will also double-check that actions (e.g., `consumeInsightToken`) are properly extracted alongside state properties without triggering global re-renders.

## 2024-05-18 - Replacing `useWorldState` with `useShallow` for Performance
**Learning:** In the `useGameStore`, `useWorldState()` is a global subscription that causes components to re-render whenever *any* part of the global state changes. Several widget and UI components were using this unnecessarily.
**Action:** Replace `useWorldState()` with `useShallow()` and explicitly select only the state properties needed by the component. This drastically reduces unnecessary re-renders in highly dynamic parts of the application (like dashboards and overlay components).
