🎯 **What:**
The `ChartStyle` component in `src/components/ui/chart.tsx` was vulnerable to potential DOM-based Cross-Site Scripting (XSS). It was injecting CSS strings (derived from `id` and `config` inputs) directly into the DOM using React's `dangerouslySetInnerHTML` prop on a `<style>` tag. While it included a custom `.replace()` method to escape `<` and `>` characters, using `dangerouslySetInnerHTML` inherently bypassed React's built-in text protections and threw ESLint security warnings.

⚠️ **Risk:**
If an attacker managed to inject malicious payloads into chart `id`s, `theme`, or `color` configuration values (e.g., `] { } </style><script>alert(1)</script>`), the browser could evaluate the injected string as executable HTML/JavaScript. Although the original code attempted to mitigate this by replacing `<` and `>`, relying on `dangerouslySetInnerHTML` when standard text children would suffice is an unnecessary risk that could lead to script execution in the context of the user's session.

🛡️ **Solution:**
I refactored the `<style>` tag to no longer use `dangerouslySetInnerHTML`. Instead, the generated CSS string is passed directly as the `children` of the `<style>` tag.

Crucially, React does *not* automatically HTML-escape text children inside `<style>` tags during Server-Side Rendering (SSR) because browsers expect literal characters (like the `>` child selector) in CSS. Therefore, I retained the original, necessary escaping logic (`.replace(/</g, "\\3C ").replace(/>/g, "\\3E ")`) to ensure that strings cannot break out of the style block during SSR, securing both the client-side injection and server-side rendering pipelines against XSS.
